
# PRD_01 — 냉장고 이미지 인식 (재료 추출)

## 1. 개요
냉장고 내부 사진을 업로드하면, `nvidia/nemotron-nano-12b-v2-vl:free` 모델을 이용해 사진 속 식재료 목록을 추출한다. 이 단계의 출력은 2단계(레시피 생성)의 입력으로 사용된다.

## 2. 목표
- 사용자가 냉장고 사진 1장을 업로드하면, 인식된 재료명 목록을 반환한다.
- 이후 단계(레시피 생성, 저장)에서 그대로 사용할 수 있는 **구조화된 형식**으로 결과를 만든다.

## 3. 범위

### In Scope
- 단일 이미지 업로드 → 재료 목록 추출 API
- 이미지 → base64 인코딩 → OpenRouter Vision 모델 호출
- 모델의 자연어 응답을 재료 배열(JSON)로 정규화
- 인식 실패/저품질 이미지에 대한 기본 에러 처리

### Out of Scope
- 레시피 생성 (→ PRD_02)
- 사용자 계정/저장 기능 (→ PRD_03)
- 다중 이미지 업로드, 동영상 인식
- 재료 수량/유통기한 추정

## 4. 사용자 플로우
1. 사용자가 웹 UI에서 냉장고 사진을 업로드한다.
2. 프론트엔드가 이미지를 백엔드 API로 전송한다 (`multipart/form-data`).
3. 백엔드가 이미지를 base64로 인코딩해 OpenRouter Vision 모델에 요청한다.
4. 모델 응답(자연어 설명 또는 목록)을 파싱해 재료명 배열로 변환한다.
5. 프론트엔드는 인식된 재료 목록을 사용자에게 보여주고, 필요시 수동으로 항목을 추가/삭제할 수 있게 한다.

## 5. API 설계

### `POST /api/vision/recognize`
**Request** (`multipart/form-data`)
| 필드 | 타입 | 설명 |
|---|---|---|
| `image` | file | jpg/png/webp, 최대 5MB |

**Response** `200 OK`
```json
{
  "ingredients": ["계란", "우유", "당근", "양파"],
  "rawModelResponse": "이 냉장고에는 계란, 우유, 당근, 양파가 보입니다."
}
```

**Error Response** `4xx`
```json
{ "error": "지원하지 않는 이미지 형식입니다." }
```

## 6. 프롬프트 설계
모델이 자유 서술형으로 응답하면 파싱이 불안정해지므로, 아래와 같이 **출력 형식을 명시**한 프롬프트를 사용한다.

```
이 이미지는 냉장고 내부 사진이다.
사진에서 식별 가능한 식재료 이름만 한국어로, 쉼표로 구분된 목록으로 답해라.
설명이나 다른 문장은 포함하지 마라.
예: 계란, 우유, 당근, 양파
```

파싱 실패(쉼표 목록 형식이 아닌 응답) 시, 응답 전체 텍스트를 `rawModelResponse`로 보존하고 `ingredients`는 best-effort로 추출한다.

## 7. 기술 의존성
- `config.js` — `OPENROUTER_API_KEY`, 모델명 (`nvidia/nemotron-nano-12b-v2-vl:free`) 검증 및 로드 (기존 구현 재사용)
- `src/services/openrouterClient.js`의 `recognizeImage(buffer, mimeType, prompt)` 재사용
- 신규: 이미지 업로드 처리(Express + multer 등), 응답 파싱 유틸

## 8. 비기능 요구사항
- 이미지 크기 제한: 5MB
- 지원 포맷: jpg, jpeg, png, webp
- 평균 응답 시간: 10초 이내 (무료 모델 특성상 지연 가능성 있음, UI에 로딩 표시 필요)
- API 키는 서버에서만 사용, 클라이언트에 노출 금지

## 9. 완료 기준 (Acceptance Criteria)
- [ ] 실제 냉장고 사진 업로드 시 재료명 3개 이상 정확히 추출
- [ ] 지원하지 않는 파일 형식/크기 초과 시 명확한 에러 메시지 반환
- [ ] 모델 응답이 예상 형식이 아니어도 서버가 크래시하지 않고 `rawModelResponse`를 반환
- [ ] `ingredients` 배열이 2단계(PRD_02) 입력 형식과 호환됨

## 10. 오픈 이슈
- 재료명 표준화(예: "계란"/"달걀" 통일) 필요 여부
- 인식 신뢰도(confidence) 값이 필요한지, 필요하다면 어떻게 획득할지
- 동일 재료 중복 인식 시 제거 로직 필요 여부
